# ZIP入力とDXFダウンロード改善 設計

## 1. 目的

通常のHPGLファイルに加えてZIPアーカイブを入力できるようにする。
ZIP内の対応HPGLをファイル単位のDXFレイヤーとして扱い、既存のプレビュー、差分表示、DXF変換へ接続する。

DXF変換後は自動的にダウンロードを開始する。
結果表示と再ダウンロード操作は残し、自動ダウンロードに失敗した場合も生成済みDXFを失わない構成とする。

## 2. 対象範囲

今回の対象は次の3項目とする。

- ZIPアップロードへの対応
- 「DXFに変換」操作後の自動ダウンロード
- 最初に正常追加された元ファイル名によるDXF既定名の設定

PDF出力は対象外とし、`docs/todo.md`のbacklogへ残す。
HPGL解析規則、ByLayerのR2000 DXF出力、プレビュー図形、差分判定規則は変更しない。

## 3. 入力の扱い

### 3.1 対応する元ファイル

ファイル選択とドラッグ＆ドロップでは、既存のHPGL拡張子と`.zip`を同時に受け付ける。
通常HPGLとZIPを一度に選択した場合は、ブラウザから受け取った元ファイル順に処理する。

ZIP内はサブフォルダを含めて検索する。
既存の対応HPGL拡張子を持つファイルだけを展開対象とし、ディレクトリ、非対応ファイル、ZIP内ZIPは無視する。

### 3.2 ZIP内ファイルの名前

ZIP内ファイルは正規化した相対パスを論理名として保持する。
たとえば`parts/A.H01`は、一覧と診断で`parts/A.H01`と表示する。

DXFレイヤー名は既存の`assignLayerNames`へ論理名を渡して生成する。
そのため、`parts/A.H01`は`parts_A`となり、同じレイヤー名が生じた場合は既存規則によって`_2`以降の接尾辞を付ける。

相対パスの区切りは`/`へ統一する。
先頭の`./`は除去し、空のパス要素と`.`は正規化する。
`..`、絶対パス、ドライブ文字、NUL文字、制御文字を含むエントリーは不正パスとして除外する。

### 3.3 入力レコード

アプリ内部では、通常ファイルとZIP内ファイルを次の入力レコードへ統一する。

```js
{
  name,       // 一覧、診断、レイヤー名に使う論理名
  blob,       // 読み込み対象のFileまたはBlob
  size,
  identity,   // 重複判定用の安定した識別子
}
```

通常ファイルの`name`は`File.name`とする。
ZIP内ファイルの`name`は正規化した相対パスとし、`blob`には展開済みバイト列を保持する。

通常ファイルの識別子は、現在のファイル名、サイズ、更新日時による規則を維持する。
ZIP内ファイルの識別子は、元ZIPの識別子とエントリー相対パスを組み合わせる。
同じZIPを再投入した場合は重複として除外し、異なるZIPに同名エントリーがある場合は両方を追加してレイヤー名で区別する。

Workerクライアントは入力レコードから構造化複製可能な`{ name, blob }`を作り、変換WorkerとプレビューWorkerへ渡す。
Workerは`blob.arrayBuffer()`で内容を読み、既存のHPGL解析へ`name`とバイト列を渡す。
この境界により、HPGL解析とDXF生成はZIP形式を認識しない。

## 4. ZIP展開

### 4.1 モジュール構成

`src/files/zip-reader.js`を新設し、ZIP展開とエントリー選別を担当させる。
ZIP展開には`fflate`の非同期APIを使用し、必要な機能だけをES Moduleとして読み込む。

ZIP展開モジュールは、元ZIPと上限設定を受け取り、展開済み入力レコード、無視した件数、警告内容を返す。
破損、暗号化、未対応圧縮方式、上限超過は、呼び出し側が分類できるエラーとして返す。

`src/files/file-policy.js`には、ZIP名の判定、元ファイル名からDXF既定名を作る純粋関数、入力レコードの重複判定に必要な処理を置く。
ZIPバイト列の解析は`file-policy.js`へ置かない。

### 4.2 展開上限

通常規模の利用を対象に、ZIPごとに次の上限を設ける。

- ZIP本体：50 MiB以下
- 対応HPGLエントリー：100件以下
- 展開後の1エントリー：20 MiB以下
- 対応HPGLの展開後合計：100 MiB以下

上限判定にはZIPエントリーの展開後サイズを使用する。
上限を超える対応エントリーが1件でもある場合は、一部だけを追加せず、そのZIP全体を失敗として扱う。
同じ相対パスを持つエントリーがZIP内に複数ある場合も、上書きによる欠落を避けるためZIP全体を失敗として扱う。

対応拡張子ではないエントリーは展開前に除外する。
これにより、画像やPDFなどの非対象データへ展開メモリを使わない。

### 4.3 非同期状態

アプリの状態へ`importing`とZIP展開ジョブを追加する。
展開中はファイル選択、ドロップ、削除、変換を無効化し、状態欄へ「ZIPを展開しています」と表示する。

同じ選択操作に含まれる元ファイルは順番に処理し、入力一覧の順序を元ファイル順とZIP内エントリー順で安定させる。
アプリ破棄時は実行中の展開を終了し、破棄後または古い処理から届いた結果を画面へ反映しない。

## 5. 部分成功とエラー表示

複数の元ファイルを同時に投入した場合は、元ファイルごとに成功または失敗を判定する。
1つのZIPが失敗しても、正常な通常HPGLとほかのZIPから得たファイルは追加する。

ZIPごとの扱いは次のとおりとする。

- 破損ZIP、暗号化ZIP、未対応圧縮方式：ZIP全体を追加せずエラー表示
- 上限超過、重複パス：ZIP全体を追加せずエラー表示
- 対応HPGLが0件：ファイルを追加せず警告表示
- 非対応ファイル、ディレクトリ、ZIP内ZIP、不正パス：対象を無視し、種類別件数をまとめて警告表示
- 既存入力との重複：重複したエントリーだけを追加せず、件数をまとめて警告表示

同じ投入操作で複数の通知が生じた場合は、元ファイル単位で内容をまとめる。
エントリーごとの長い警告列挙は行わない。

## 6. DXF既定名

初期表示は現在と同じ`converted.dxf`とする。
最初に1件以上の入力レコードを正常追加できた元ファイルから、一度だけ既定名を設定する。

通常HPGLが元ファイルの場合は、最後の拡張子を`.dxf`へ置換する。
たとえば`A.H01`は`A.dxf`となる。
ZIPが元ファイルの場合も同じ規則を使い、`drawings.zip`は`drawings.dxf`となる。

破損ZIP、対応HPGLが0件のZIP、非対応ファイルは既定名の決定対象にしない。
同じ投入操作の先頭が失敗した場合は、その後で最初に正常追加できた元ファイルを使用する。

ユーザーが最初の正常追加より前に出力名を編集した場合は、その入力値を上書きしない。
最初の正常追加を処理した後は、入力ファイルの追加、削除、全削除によって出力名を自動変更しない。

## 7. 自動ダウンロード

変換WorkerがDXFを正常生成した時点で、現在の出力名を正規化してダウンロードを1回開始する。
診断にエラーが含まれる場合でも、変換WorkerがDXF結果を返したなら自動ダウンロードする。

自動ダウンロード後も、変換結果、診断、再ダウンロードボタンを表示する。
再ダウンロードは保持済みの同じDXFバッファを使う。

Blob URLの作成、非表示アンカーのクリック、URL破棄は一つのダウンロード関数へ集約する。
自動ダウンロード開始に失敗した場合は変換結果を維持し、再ダウンロードボタンを案内する警告を表示する。
ダウンロード失敗をDXF変換失敗として扱わない。

変換完了ハンドラーは結果表示を確定してから自動ダウンロードを呼び出す。
同じ変換結果について、非同期完了通知の重複や再描画によって自動ダウンロードを再実行しない。

## 8. 画面表示

入力案内とファイル選択の`accept`へ`.zip`を追加する。
プライバシー表示には、ZIPもブラウザのメモリ内だけで展開することを反映する。

ZIP展開後の一覧は通常HPGLと同じ列を使用する。
ZIP内ファイルのファイル名列には相対パスを表示し、元ZIP名を別列として追加しない。

展開中は既存の変換進捗と混同しない状態文を表示する。
ファイル一覧と出力設定の基本配置は変更しない。

## 9. テスト方針

### 9.1 ZIP展開

- ZIP直下とサブフォルダにある対応HPGLの抽出
- ZIPエントリー順の保持
- 日本語を含む相対パス
- 非対応ファイル、ディレクトリ、ZIP内ZIPの除外
- `..`、絶対パス、ドライブ文字、NUL文字、制御文字を含む不正パスの除外
- 破損ZIP、暗号化ZIP、未対応圧縮方式の失敗
- ZIP本体、エントリー数、1件の展開後サイズ、展開後合計の上限
- ZIP内の重複相対パス
- 同じZIPの再投入と、異なるZIPにある同名エントリー
- 対応HPGLが0件のZIP

テスト用ZIPはテストコード内で生成し、実ZIPのバイナリをリポジトリへ追加しない。
上限テストでは設定値を注入し、小さなデータで境界値を検証する。

### 9.2 UI状態

- 通常HPGL、ZIP、両者を混在させた追加順
- ZIP内相対パスの一覧表示とレイヤー名
- 1つのZIPが失敗した場合の部分成功
- 展開中の入力、削除、変換の無効化
- アプリ破棄後と古い処理結果の無視
- ZIP名と通常HPGL名からのDXF既定名
- 失敗した先頭元ファイルを既定名に使わないこと
- アップロード前の手入力を保持すること
- 追加、削除、全削除後に既定名を自動変更しないこと

### 9.3 ダウンロード

- 変換成功時に1回だけ自動ダウンロードすること
- 診断エラーを含む生成結果も自動ダウンロードすること
- 結果欄と再ダウンロードボタンを残すこと
- 再ダウンロードで同じ結果と正規化済みファイル名を使うこと
- ダウンロード開始失敗後も結果と再ダウンロード操作を残すこと
- Blob URLを成功時と失敗時の双方で解放すること

### 9.4 回帰と全体検証

- 通常HPGLだけを追加する既存UIテスト
- プレビューと差分表示
- 変換WorkerとプレビューWorkerの読み込み失敗処理
- ByLayerのR2000 DXF構造
- 参照HPGL群の図形数とDXFフィンガープリント
- `npm.cmd test`
- `npm.cmd run build`
- `npm.cmd run generate:reference-dxf`
- `git diff --check`

## 11. 完了条件

- 通常HPGLとZIPを同時に追加できる。
- ZIP内のサブフォルダから対応HPGLだけを安全上限内で抽出できる。
- ZIP内ファイルの相対パスが一覧、診断、レイヤー名へ一貫して反映される。
- 1つのZIPが失敗しても、同時投入した正常ファイルを利用できる。
- 最初に正常追加された元ファイルからDXF既定名を一度だけ設定できる。
- 変換成功時にDXFを1回自動ダウンロードし、結果欄から再ダウンロードできる。
- 通常HPGL、プレビュー、差分表示、ByLayerのR2000 DXF出力に回帰がない。
- 自動テスト、ビルド、参照DXF生成、差分検査を完了する。
